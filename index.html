Introduction

Pink Dreams is a complete commerce platform with two synchronized layers: customer storefront and administrative control panel.

The platform is production-oriented: it supports multi-payment checkout, order workflows, receipt verification, role-based access control, real-time admin notifications, and dynamic settings.

Business positioning: This project is suitable for direct commercial resale, client delivery, or white-label adaptation.

Architecture Overview

The system follows a decoupled API architecture:

  • Frontend: Next.js App Router, client/server rendering mix, context-based app state.
  • Backend: Express REST APIs with modular routes/controllers/middleware.
  • Database: MongoDB with Mongoose schemas for domain models.
  • Realtime: Socket.IO for admin notification events.
  • Media: Local upload + cloud integration support (Cloudinary-ready patterns).

Local defaults: Frontend: http://localhost:3000 | Backend: http://localhost:4000

Run on Live/Production Server

  1. Deploy backend first with production env vars and managed MongoDB.
  2. Deploy frontend and set `NEXT_PUBLIC_API_URL` to backend public URL.
  3. Set OAuth callback URLs to live domain.
  4. Set CORS origins to production domain list only.
  5. Enable HTTPS and strict session/cookie security in production.
Important: Never deploy with default secrets or localhost URLs in production.

Authentication Flow

Platform supports email/password and OAuth providers. Auth state is managed in frontend context.

  • JWT token is issued by backend and stored client-side.
  • Route guards decide if modal login is required by route policy.
  • Protected APIs use `Authorization: Bearer <token>`.
  • Token validation failures trigger clean logout state.

Staff Roles & Permissions

Admin module uses permission-based authorization for staff users. Example permissions: `products:read`, `orders:read`, `analytics:read`, `notifications:read`.

  • Roles are centrally managed and assigned to team members.
  • UI elements are conditionally rendered by permission.
  • Sensitive actions are also protected at API level.

Orders Lifecycle

The order lifecycle is standardized for operational consistency.

  1. pending – Order created
  2. confirmed – Validated by system/admin
  3. processing – Preparing dispatch
  4. shipped – In transit
  5. delivered – Completed
  6. cancelled – Terminated safely

Orders page includes search, status filter, pagination, and modal details for full context.

Security Hardening

  • Use strong JWT/session secrets per environment
  • Apply strict CORS origin whitelist
  • Enable HTTPS-only cookies in production
  • Validate all input payloads on backend
  • Restrict admin routes with role/permission middleware
  • Rotate API keys and OAuth secrets periodically

Performance & Scaling

  • Use caching for public settings and frequently read queries
  • Optimize image delivery and enforce compressed formats
  • Lazy load non-critical client components
  • Use DB indexes on high-traffic fields (orderId, status, createdAt)
  • Use process manager (PM2/systemd) with health probes

Backup & Recovery

  • Automated daily database backups
  • Weekly restore verification on staging
  • Export key business datasets (orders, products, users)
  • Document emergency rollback steps per deployment

Monitoring & Logs

  • Monitor API error rates and slow routes
  • Capture authentication and payment failures
  • Track notification/socket connection health
  • Use structured logs for searchable incident analysis

SEO & Content Strategy

  • Use clean metadata for homepage, product, category, blog pages
  • Publish regular blog content with target keywords
  • Optimize product images and structured content
  • Use internal links between blog and commerce pages

Troubleshooting

Issue Likely Cause Fix
Orders page fails Missing/invalid token Check auth storage, protect routes, verify backend token middleware
Order details fail after payment Wrong endpoint or missing auth header Use authenticated `/orders/:orderId` calls
MIME stylesheet error Wrong CSS path/name Ensure `index.html` uses exact existing css file path
Image not rendering Broken URL normalization Verify `NEXT_PUBLIC_API_URL` + image utility fallback logic
